# Техническое задание

## 1. Название проекта
Интерактивный образовательный веб-проект о криптографии и Холодной войне.

Рабочее название: **Cold War: History**.

## 2. Идея проекта
Проект объединяет три направления в одном сайте:

1. Образовательный контент о шифровании и историческом контексте.
2. Интерактивная лаборатория, где пользователь кодирует и декодирует сообщения разными методами.
3. Игровые режимы (включая сценарии в духе Papers, Please), где нужно расшифровывать, анализировать и принимать решения.

## 3. Цель
Сделать сайт, который:

1. Понятно объясняет, как работают шифры и зачем они были важны в истории.
2. Дает возможность практиковаться в шифровании/дешифровании прямо в браузере.
3. Удерживает внимание через игровые механики, прогресс и соревновательные элементы.

## 4. Ограничения и базовые допущения

1. Архитектура: **микросервисная**.
2. Технологический стек: **не фиксирован** (выбирается командой отдельно).
3. Проект должен поддерживать поэтапную реализацию: MVP -> расширенная версия.

## 5. Роли пользователей

1. Гость: просматривает часть контента и использует базовые интерактивы.
2. Пользователь: имеет аккаунт, сохраняет прогресс, играет в режимы, участвует в рейтингах.
3. Редактор: создает и правит исторические материалы, карточки шифров, задания.
4. Администратор: управляет пользователями, контентом, сложностью и настройками игровых режимов.

## 6. Функциональные требования

### 6.1. Контент о шифровании

1. **FR-01**: Сайт должен показывать каталог шифров с карточками (название, эпоха, краткая суть, уровень сложности).
2. **FR-02**: Для каждого шифра должна быть отдельная страница с историей, принципом работы и примером.
3. **FR-03**: Пользователь должен видеть пошаговое объяснение алгоритма шифрования на конкретном примере.
4. **FR-04**: Сайт должен поддерживать фильтрацию шифров по типу (подстановка, перестановка, полиалфавитные и т.д.).
5. **FR-05**: Сайт должен поддерживать поиск по названию шифра, эпохе и ключевым словам.
6. **FR-06**: На странице шифра должны отображаться связанные исторические события.
7. **FR-07**: Сайт должен хранить версии материалов (кто и когда обновил описание).
8. **FR-08**: Должна быть возможность отметить карточку шифра как избранную (для авторизованных пользователей).

### 6.2. Криптографическая лаборатория

1. **FR-09**: Пользователь должен выбирать шифр из списка и вводить произвольное текстовое сообщение.
2. **FR-10**: Система должна поддерживать режимы "Зашифровать" и "Расшифровать".
3. **FR-11**: Для шифров с параметрами пользователь должен задавать ключ/сдвиг/другие параметры.
4. **FR-12**: Система должна валидировать входные данные и показывать понятные ошибки.
5. **FR-13**: Результат преобразования должен отображаться сразу после запуска операции.
6. **FR-14**: Пользователь должен иметь кнопку "Поменять местами" для быстрого перехода от шифрования к дешифровке.
7. **FR-15**: Система должна показывать краткое объяснение, почему получился именно такой результат.
8. **FR-16**: Пользователь должен иметь возможность копировать результат в буфер обмена.
9. **FR-17**: Для авторизованного пользователя должна сохраняться история последних операций.
10. **FR-18**: История операций должна поддерживать повторный запуск (replay) с теми же параметрами.

### 6.3. Тренировочный режим

1. **FR-19**: Пользователь должен запускать тренировочные задания по выбранному шифру.
2. **FR-20**: Система должна автоматически генерировать задания разной сложности.
3. **FR-21**: Пользователь должен получать проверку ответа и объяснение ошибки.
4. **FR-22**: Система должна начислять очки за правильные ответы.
5. **FR-23**: Должна быть система подсказок с штрафом по очкам.
6. **FR-24**: Должна отображаться статистика: точность, среднее время, лучший результат.
7. **FR-25**: Должны быть ежедневные задания (Daily Challenge).

### 6.4. Игровой режим "Инспектор связи" (в духе Papers, Please)

1. **FR-26**: Пользователь получает роль оператора/аналитика и поток входящих сообщений.
2. **FR-27**: Для каждого сообщения пользователь должен принять решение (пропустить, отклонить, эскалировать).
3. **FR-28**: Перед решением пользователь должен расшифровать или проверить достоверность сообщения.
4. **FR-29**: Игра должна учитывать время реакции и точность решения.
5. **FR-30**: Должны быть игровые последствия ошибок (потеря очков, ухудшение рейтинга, сюжетные развилки).
6. **FR-31**: Система должна генерировать смены/уровни с растущей сложностью.
7. **FR-32**: Должны присутствовать события-сюрпризы (ложные сигналы, двойные сообщения, конфликтующие приказы).
8. **FR-33**: После смены пользователь должен получать отчет по ошибкам и рекомендации.

### 6.5. Исторический модуль (Холодная война)

1. **FR-34**: Сайт должен содержать интерактивную временную шкалу ключевых событий.
2. **FR-35**: Каждое событие должно иметь карточку: дата, участники, описание, последствия.
3. **FR-36**: Пользователь должен фильтровать события по региону, году, теме (конфликты, вооружение, разведка, дипломатия).
4. **FR-37**: Должны быть тематические подборки (например: Карибский кризис, гонка вооружений, разведоперации).
5. **FR-38**: Исторические события должны быть связаны с соответствующими шифрами и игровыми заданиями.

### 6.6. Аккаунт, прогресс и соревновательные механики

1. **FR-39**: Пользователь должен регистрироваться, входить и выходить из аккаунта.
2. **FR-40**: Должен сохраняться персональный прогресс по модулям (лаборатория, тренировки, игра).
3. **FR-41**: Должны быть достижения (ачивки) за конкретные действия.
4. **FR-42**: Должна быть таблица лидеров по неделе и по общему рейтингу.
5. **FR-43**: Пользователь должен видеть свой профиль со статистикой и историей активности.

### 6.7. Админ-панель и управление контентом

1. **FR-44**: Администратор/редактор должен создавать, редактировать и архивировать карточки шифров.
2. **FR-45**: Администратор/редактор должен управлять историческими событиями и тематическими подборками.
3. **FR-46**: Администратор должен настраивать параметры игровых режимов (сложность, таймеры, веса очков).
4. **FR-47**: Администратор должен иметь журнал действий (кто и что менял).
5. **FR-48**: Администратор должен управлять публикацией контента (черновик/опубликовано).

### 6.8. Функциональные требования к микросервисам

1. **FR-49**: Система должна использовать API Gateway как единую точку входа для клиента.
2. **FR-50**: Должен быть отдельный сервис аутентификации и авторизации.
3. **FR-51**: Должен быть отдельный сервис криптоопераций (шифрование/дешифрование/валидация).
4. **FR-52**: Должен быть отдельный сервис контента (шифры, исторические материалы, таймлайн).
5. **FR-53**: Должен быть отдельный игровой сервис (режимы, задания, проверка результатов).
6. **FR-54**: Должен быть отдельный сервис профиля и прогресса пользователя.
7. **FR-55**: Сервисы должны обмениваться событиями (например, "задание выполнено", "ачивка получена").
8. **FR-56**: Должно быть централизованное логирование пользовательских действий и игровых результатов.

## 7. Состав MVP (рекомендуемая первая версия)

1. Каталог из 5-8 шифров с карточками и примерами.
2. Лаборатория шифрования/дешифрования для этих шифров.
3. Базовый тренировочный режим с автоматической проверкой.
4. Базовый исторический раздел: таймлайн из 15-20 событий.
5. Авторизация и сохранение истории операций пользователя.
6. Черновой игровой режим "Инспектор связи" в формате 1 сценария.

## 8. Идеи расширения (чтобы проект был "не маленьким" для команды из 3 человек)

1. Кооперативный режим на 2-3 игроков: один перехватывает, второй расшифровывает, третий принимает решение.
2. Редактор пользовательских миссий: игроки создают и публикуют свои задания.
3. Модуль "Стеганография": скрытие сообщений в изображениях/тексте и разбор таких кейсов.
4. Интерактивная карта мира с геопривязкой операций и конфликтов.
5. Сюжетная кампания по эпохам (1946-1953, 1953-1962, 1962-1979, 1979-1991).
6. Режим "дебаты аналитиков": выбрать интерпретацию разведданных и увидеть исторический исход.
7. Система сезонных событий (ежемесячные наборы миссий и рейтинг).
8. Раздел "разбор ошибок": персональные рекомендации, что изучить дальше.

## 9. Критерии приемки

1. Пользователь может выбрать шифр, зашифровать текст и успешно расшифровать его обратно.
2. Для каждого шифра доступны описание, пример и исторический контекст.
3. Тренировочные задания проверяются автоматически и сохраняют результат.
4. Игровой режим выдает итог смены с очками и объяснением ошибок.
5. Исторический раздел содержит фильтруемый таймлайн и связь с криптографическими материалами.
6. Прогресс авторизованного пользователя сохраняется между сессиями.

## 10. Открытые вопросы на уточнение

1. Будет ли регистрация обязательной для игры или только для сохранения прогресса?
2. Какие шифры входят в первую версию (фиксированный список)?
3. Нужен ли мультиплеер в рамках текущего этапа или это пост-MVP?
4. Нужна ли модерация пользовательского контента (если появится редактор миссий)?
5. Нужен ли только русский язык интерфейса или сразу RU + EN?
6. Есть ли ограничения по хостингу/бюджету на инфраструктуру микросервисов?

## 11. Разделение задач по команде (фронт, бэк, БД)

### 11.1. Задачи Frontend

1. Сделать структуру страниц и роутинг: главная, каталог шифров, страница шифра, лаборатория, тренировки, игровой режим, таймлайн, профиль, вход/регистрация, админка.
2. Реализовать UI каталога шифров с поиском и фильтрами по типу/эпохе/сложности.
3. Реализовать страницу конкретного шифра с историей, принципом работы и пошаговым примером.
4. Реализовать форму криптолаборатории: поле текста, выбор режима (шифровать/дешифровать), параметры шифра, вывод результата, кнопка обмена входа/выхода.
5. Добавить клиентскую валидацию ввода и понятные сообщения об ошибках.
6. Реализовать UI тренировочного режима: задача, ввод ответа, подсказка, проверка, таймер, начисление очков.
7. Реализовать UI игрового режима "Инспектор связи": поток сообщений, окно анализа, кнопки решений, отчет по смене.
8. Сделать интерактивный таймлайн с фильтрами по региону/году/теме и переходами в связанные материалы.
9. Реализовать личный кабинет: история операций, прогресс, ачивки, статистика, рейтинг.
10. Реализовать админский интерфейс для CRUD-операций с шифрами, событиями и заданиями.
11. Настроить единый API-клиент к Gateway (обработка токена, ошибок, повторных запросов при истечении сессии).
12. Обеспечить адаптивность (desktop/tablet/mobile) и базовую доступность (контраст, фокус, навигация с клавиатуры).

### 11.2. Задачи Backend

1. Поднять API Gateway как единую точку входа и маршрутизацию запросов по сервисам.
2. Реализовать сервис аутентификации/авторизации: регистрация, логин, refresh-токены, роли (гость/пользователь/редактор/админ).
3. Реализовать криптосервис: операции шифрования/дешифрования, валидация параметров, возвращение пояснений по шагам.
4. Реализовать сервис контента: CRUD для карточек шифров, исторических событий, подборок, статусы публикации.
5. Реализовать игровой сервис: генерация заданий, проверка ответов, расчет очков, финальный отчет по смене.
6. Реализовать сервис прогресса: история операций, достижения, метрики пользователя, данные для таблицы лидеров.
7. Реализовать API ежедневных заданий и планировщик их обновления.
8. Внедрить событийное взаимодействие между сервисами (например: "challenge.completed", "achievement.unlocked").
9. Ввести централизованный аудит действий (админ-изменения, ключевые игровые события, попытки входа).
10. Настроить ограничения доступа по ролям для пользовательских и административных эндпоинтов.
11. Подготовить OpenAPI/Swagger по всем внешним эндпоинтам.
12. Написать интеграционные и контрактные тесты между Gateway и микросервисами.

### 11.3. Задачи Database

1. Спроектировать модель хранения в формате "БД на сервис" (отдельная БД/схема для auth, content, game, progress).
2. Подготовить ER-диаграммы и схему связей сущностей для каждого сервиса.
3. Сформировать таблицы пользователей, ролей, сессий и токенов для auth-сервиса.
4. Сформировать таблицы карточек шифров, версий материалов, исторических событий и подборок для content-сервиса.
5. Сформировать таблицы заданий, попыток, смен, игровых решений и итогов для game-сервиса.
6. Сформировать таблицы операций шифрования, статистики, достижений и лидерборда для progress-сервиса.
7. Подготовить миграции и правила версионирования схемы (up/down) для всех сервисов.
8. Настроить индексы под ключевые запросы: поиск по шифрам, фильтры таймлайна, лидерборд, история операций.
9. Подготовить seed-данные для MVP: набор шифров, примеры, 15-20 исторических событий, стартовые задания.
10. Настроить резервное копирование и восстановление для каждой сервисной БД.
11. Настроить хранение и ротацию больших журналов (аудит, история операций) с политикой ретенции.
12. Подготовить мониторинг БД: медленные запросы, рост таблиц, состояние индексов.
